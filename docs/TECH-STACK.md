# 技術スタック — UniteCube ビジネス自動化サービス

_作成日：2026年4月22日 / v1.0_

---

## 概要

本ドキュメントは、UniteCubeが採用する技術スタックの選定理由、トレードオフ、および検討した代替案を記録します。意思決定の根拠を残すことで、将来の技術見直しや新メンバーへの引き継ぎを容易にします。

### 判断基準

技術選定において以下の優先順位で評価しています。

1. **チームの既存スキルとの整合性** — 学習コストを最小化し、即戦力として使える
2. **運用コストの低さ** — 月額費用を抑え、利益率を確保する（目標：月商20万円以上）
3. **クライアント接続性** — 北海道の中小企業が日常的に使うツール（LINE、Gmail、Googleスプレッドシート）と確実に連携できる
4. **セルフホスト可能性** — SaaSへの依存を減らし、価格改定・サービス終了リスクを軽減する
5. **小規模チームでの保守性** — 複雑な運用手順なしに1〜2人で維持できる

## 採用技術スタック

### 1. ワークフロー自動化：n8n

| 項目 | 内容 |
|------|------|
| **役割** | 業務プロセスの自動化エンジン（メール通知、予約管理、レポート生成など） |
| **ライセンス** | Sustainable Use License（セルフホスト版は無料） |
| **ホスト形態** | VPS上にDockerで自社ホスト |

**選定理由**

- 既存チームがn8nの操作・開発に習熟している
- 400以上のコネクタを標準搭載し、LINE・Gmail・Googleスプレッドシートなどクライアントが使うツールに追加開発なしで接続できる
- セルフホストにより月額ツール費用がほぼゼロ（VPS費用のみ）
- Webhookベースのトリガーで低レイテンシな自動化を実現できる

**トレードオフ**

- サーバ障害時は自社で対応が必要（クラウド版は自動冗長化あり）
- n8n自体のバージョンアップ管理が必要

**検討した代替案**

| 代替案 | 却下理由 |
|--------|---------|
| Zapier | 月額費用が高く（$20〜$100+/月）、複数クライアントへのスケールでコストが急増する |
| Make（旧Integromat） | 機能は近いが習熟度がなく学習コスト増。有料プランが必要な場合が多い |
| AWS Step Functions | 開発・運用難易度が高くチームの現スキルセットに合わない |

### 2. AIワークフロー：Dify

| 項目 | 内容 |
|------|------|
| **役割** | LLMを使った高度な自動応答・文書生成・チャットボット構築 |
| **ライセンス** | Apache 2.0（コミュニティ版） |
| **ホスト形態** | VPS上にDockerで自社ホスト |

**選定理由**

- 既存チームがDifyを使った開発経験を持つ
- プロンプトのGUI編集・バージョン管理ができ、非エンジニアでも運用調整が可能
- n8nとWebhook/API経由で連携し、AIタスクをワークフローに組み込める
- セルフホストによりAPI呼び出しコストのみ（Dify本体費用なし）

**トレードオフ**

- LLM APIの呼び出しコスト（OpenAI / Anthropicなど）は別途発生する
- モデル性能はAPI側の制約に依存する

**検討した代替案**

| 代替案 | 却下理由 |
|--------|---------|
| LangChain（Pythonで直書き） | 開発工数が増え、変更のたびにデプロイが必要。ノーコードの柔軟性がない |
| Flowise | 機能は類似するが、Difyのほうが本番運用実績が豊富でドキュメントが充実 |

### 3. Webサイト・カスタム開発：WordPress + PHP

| 項目 | 内容 |
|------|------|
| **役割** | コーポレートサイト、問い合わせフォーム、簡易管理画面の構築 |
| **ライセンス** | GPL v2 |
| **ホスト形態** | 共用サーバまたはVPS（KUSANAGI等の高速環境を推奨） |

**選定理由**

- チームの主力スキルであり開発・保守コストが最低
- クライアント自身が慣れているケースが多く、コンテンツ管理を引き渡しやすい
- ACF、WooCommerce、Contact Form 7など豊富なプラグインで短期間に機能追加できる

**トレードオフ**

- セキュリティアップデートの継続的な適用が必要
- プラグインの乱用によるサイト肥大化・競合リスク

**検討した代替案**

| 代替案 | 却下理由 |
|--------|---------|
| Laravel | モダンだがWordPressよりも構築工数が増える。既存クライアントとの整合性も低い |
| Wix / Squarespace | エクスポート・移植が困難でベンダーロックインが強い |

### 4. バックエンドスクリプト・データ処理：Python

| 項目 | 内容 |
|------|------|
| **役割** | データ変換、バッチ処理、n8nから呼び出す汎用スクリプト、Dify連携の補助処理 |
| **ランタイム** | Python 3.11+ |
| **主要ライブラリ** | `requests`, `pandas`（データ整形）, `openpyxl`（Excel出力）, `python-dotenv` |

**選定理由**

- チームが習熟しており、特にデータ加工・APIラッパー作成で即時生産性を発揮できる
- n8nの「Execute Command」または「HTTP Request」ノードから容易に呼び出せる
- 豊富なライブラリで多様なクライアント要件に対応できる

**トレードオフ**

- Pythonスクリプトの実行環境管理（virtualenv / Dockerコンテナ）が必要
- 長期的な保守のためにコードレビューと最低限のテストが求められる

### 5. クライアントダッシュボード

| 項目 | 内容 |
|------|------|
| **役割** | 実行履歴の確認、ワークフローの一時停止/再開、サポートリクエスト送信 |
| **採用アプローチ** | WordPress + カスタム投稿タイプ + 制限付きログインロール（初期フェーズ） |

**選定理由**

- 初期フェーズは既存のWordPress環境を活用し、開発工数を最小化
- クライアントごとにWordPressユーザーアカウントを作成し、専用ページで実行ログを表示
- n8nのAPI（`/executions`エンドポイント）をWordPressからポーリングしてログを取得できる

**将来的な移行オプション**

ユーザー数が増えた場合、以下への移行を検討します。

- **軽量SPAフロントエンド**（Vue.js or Alpine.js）+ PHP/Python API
- **Retool / AppSmith**（ノーコード管理UI。セルフホスト可能）

### 6. インフラ・ホスティング

| 用途 | 採用技術 | 理由 |
|------|---------|------|
| n8n / Dify 実行環境 | VPS（ConoHaまたはさくらのVPS） | 国内リージョンで低レイテンシ、月額費用が安価（2,000〜5,000円/月）|
| WordPress | 共用ホスティングまたはVPS | クライアント数が少ない初期はコストを抑えられる |
| コンテナ管理 | Docker Compose | 単一サーバで複数サービスを管理する最小構成 |
| SSL | Let's Encrypt（Certbot） | 無料かつ自動更新 |
| バックアップ | rclone + オブジェクトストレージ（Wasabiなど） | 低コストで自動バックアップ |

## セキュリティ基本方針

- n8n / DifyのWebUIは**VPN経由またはIP制限**で保護し、公開インターネットに直接露出させない
- クライアントごとのAPIキーは環境変数（`.env`）で管理し、リポジトリに含めない
- WordPressには**二要素認証（2FA）**プラグインを導入
- LLM APIキー（OpenAI等）はDify側で一元管理し、クライアントには共有しない
