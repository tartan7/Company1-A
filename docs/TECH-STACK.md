# 技術スタック — UniteCube ビジネス自動化サービス

作成日：2026年4月22日 / v1.0

---

## 概要

本ドキュメントは、UniteCubeが採用する技術スタックの選定理由、トレードオフ、および検討した代替案を記録します。意思決定の根拠を残すことで、将来の技術見直しや新メンバーへの引き継ぎを容易にします。

### 判断基準

技術選定において以下の優先順位で評価しています。

1. **チームの既存スキルとの整合性** — 学習コストを最小化し、即戦力として使える
2. **運用コストの低さ**
   — 月額費用を抑え、利益率を確保する（目標：月商20万円以上）
3. **クライアント接続性**
   — 北海道の中小企業が日常的に使うツール（LINE、Gmail、Googleスプレッドシート）と確実に連携できる
4. **セルフホスト可能性** —
   SaaSへの依存を減らし、価格改定・サービス終了リスクを軽減する
5. **小規模チームでの保守性** — 複雑な運用手順なしに1〜2人で維持できる

## 採用技術スタック

### 1. ワークフロー自動化：n8n

| 項目 |
| -------------- |
| **役割** |
| **ライセンス** |
| **ホスト形態** |

- 既存チームがn8nの操作・開発に習熟している
- 400以上のコネクタを搭載し、主要ツールへ追加開発なしで接続できる
- セルフホストにより月額ツール費用がほぼゼロ（VPS費用のみ）
- Webhookベースのトリガーで低レイテンシな自動化を実現できる

- サーバ障害時は自社で対応が必要（クラウド版は自動冗長化あり）
- n8n自体のバージョンアップ管理が必要

| 代替案 |
| --- |
| Zapier |
| Make（旧Integromat） |
| AWS Step Functions |

### 2. AIワークフロー：Dify

| 項目 |
| -------------- |
| **役割** |
| **ライセンス** |
| **ホスト形態** |

- 既存チームがDifyを使った開発経験を持つ
- プロンプトのGUI編集・バージョン管理ができ、非エンジニアでも運用調整が可能
- n8nとWebhook/API経由で連携し、AIタスクをワークフローに組み込める
- セルフホストによりAPI呼び出しコストのみ（Dify本体費用なし）

- LLM APIの呼び出しコスト（OpenAI / Anthropicなど）は別途発生する
- モデル性能はAPI側の制約に依存する

| 代替案 |
| --------------------------- |
| LangChain（Pythonで直書き） |
| Flowise |

### 3. Webサイト・カスタム開発：WordPress + PHP

| 項目 |
| -------------- |
| **役割** |
| **ライセンス** |
| **ホスト形態** |

### 4. バックエンドスクリプト・データ処理：Python

| 項目 |
| ------------------ |
| **役割** |
| **ランタイム** |
| **主要ライブラリ** |

- チームが習熟しており、特にデータ加工・APIラッパー作成で即時生産性を発揮できる
- n8nの「Execute Command」または「HTTP Request」ノードから容易に呼び出せる
- 豊富なライブラリで多様なクライアント要件に対応できる

- Pythonスクリプトの実行環境管理（virtualenv / Dockerコンテナ）が必要
- 長期的な保守のためにコードレビューと最低限のテストが求められる

---

### 5. クライアントダッシュボード

| 項目 |
| --- |
| **役割** |
| **採用アプローチ** |

- 初期フェーズは既存のWordPress環境を活用し、開発工数を最小化
- クライアントごとにWordPressユーザーアカウントを作成し、専用ページで実行ログを表示
- n8nのAPI（`/executions`エンドポイント）をWordPressからポーリングしてログを取得できる

ユーザー数が増えた場合、以下への移行を検討します。

- **軽量SPAフロントエンド**（Vue.js or Alpine.js）+ PHP/Python API
- **Retool / AppSmith**（ノーコード管理UI。セルフホスト可能）

## セキュリティ基本方針

- n8n /
  DifyのWebUIは**VPN経由またはIP制限**で保護し、公開インターネットに直接露出させない
- クライアントごとのAPIキーは環境変数（`.env`）で管理し、リポジトリに含めない
- WordPressには**二要素認証（2FA）**プラグインを導入
- LLM APIキー（OpenAI等）はDify側で一元管理し、クライアントには共有しない
