## 1. 目的・適用範囲・用語定義

### 1.1 目的

本SOPは、UniteCubeプロジェクトにおける品質管理のプロセスを統一し、すべてのエージェント・チームメンバーが同一の手順・書式・基準で作業できるようにすることを目的とする。

### 1.2 適用範囲

本SOPはUniteCubeプロジェクトの以下のフェーズに適用する。ただし、フェーズが進むにつれて品質基準の粒度が細かくなる（詳細は[セクション4](#4-フェーズ別品質基準と粒度)参照）。

| フェーズ | 内容 | 品質粒度 |
|----------|------|----------|
| F1: 企画 | 要件定義・スコープ確認 | 粗（方向性・実現可能性） |
| F2: 設計 | アーキテクチャ・デザイン設計 | 中（設計整合性・インテーク基準） |
| F3: 開発 | 実装・コーディング | 細（コードレビュー・単体テスト） |
| F4: テスト | 統合・システムテスト | 細（テストケース・欠陥記録） |
| F5: デプロイ | リリース・本番反映 | 非常に細（ゲートチェックリスト・ロールバック手順） |
| F6: 保守 | 運用・モニタリング・改善 | 継続的（メトリクス・インシデント対応） |

### 1.3 用語定義

| 用語 | 定義 |
|------|------|
| 不適合 | 品質基準・受入れ基準を満たさない状態 |
| 是正処置 | 不適合の根本原因を除去するための対応 |
| 予防策 | 不適合の発生を未然に防ぐための対応 |
| 品質ゲート | 次フェーズへ進む前に必達しなければならない品質基準 |
| SOP | Standard Operating Procedure（標準作業手順） |

## 2. この手順書の改訂ルール

本SOPは生きた文書であり、定期的な見直しと改訂が必要である。

### 2.1 改訂トリガー条件

以下のいずれかが発生した場合、改訂を検討する。

- プロジェクトフェーズの追加・変更・削除
- 重大インシデントの発生（同種問題の再発防止）
- チームメンバーやエージェントから「この手順では対応不能」という報告
- 四半期レビュー（定期改訂）
- 関連文書（テスト計画書フォーマット等）の変更
- 技術スタックまたはCI/CD手順の変更

### 2.2 改訂手順

```
1. 改訂案の作成（提案者: チームメンバー / エージェント）
   ↓
2. レビュー（プロジェクトマネージャー + 影響を受けるチームリーダー）
   ↓
3. 変更箇所の差分確認（旧版との比較、下流手順書への影響確認）
   ↓
4. 承認（CEOまたはプロジェクトマネージャー）
   ↓
5. 改訂履歴テーブルの更新（バージョン番号・日付・改訂内容・作成者）
   ↓
6. 関係者への周知（Issueコメントまたは通知）
   ↓
7. 旧版のアーカイブ（git履歴で管理）
```

### 2.3 バージョン命名規則

- **メジャー改訂** (x.0): 手順・フローの大幅変更、新フェーズ追加
- **マイナー改訂** (x.y): テンプレートの追加・修正、チェックリスト項目の追加・削除
- **パッチ** (x.y.z): 誤字・リンク修正、軽微な表現修正

## 3. 責任者と承認基準

### 3.1 役割と責任

| 役割 | 担当者 | 責任範囲 | 承認権限 |
|------|--------|----------|----------|
| CEO | CEOエージェント | SOP全体の整合性・戦略的品質目標 | SOP改訂・重大不適合 |
| プロジェクトマネージャー | PMエージェント/担当者 | プロセス遵守・品質計画・メトリクス報告 | リリース判定・是正処置 |
| QAリーダー | QAエージェント/担当者 | テスト計画・実施・欠陥管理 | 品質ゲート合否判定 |
| 開発リーダー | CTOエージェント/担当者 | コード品質・技術的品質基準 | コードレビュー承認・技術設計承認 |
| デザインリーダー | デザイン担当者 | デザイン仕様遵守・ユーザビリティ | デザインレビュー承認・インテーク承認 |
| 各担当エージェント | 実装担当 | 担当タスクの品質基準遵守・テスト作成 | 自タスクの完了宣言 |

### 3.2 承認基準（各ステージ）

| ステージ | 承認者 | 承認基準 |
|----------|--------|----------|
| 要件確定 | PM + CEO | 要件が明確・実現可能・スコープ合意済み |
| 設計レビュー | 開発リーダー + デザインリーダー | 技術整合性OK・インテークチェックリストPASS |
| コードレビュー | 開発リーダー（または委任された上級エンジニア） | レビュー指摘対応済み・テストカバレッジ≥80% |
| テスト完了 | QAリーダー | 重大・高優先欠陥ゼロ・全テストケース実施済み |
| リリース判定 | PM + QAリーダー | 品質ゲート全項目PASS |
| SOP改訂 | CEO または PM | セクション2の改訂手順完了 |

## 4. フェーズ別品質基準と粒度

フェーズが後工程になるほど品質確認の粒度は細かくなる。各フェーズで何を確認すべきかを以下に示す。

### F1: 企画フェーズ

**品質の観点:** 方向性・実現可能性・リスクの大まかな把握

**確認項目:**
- [ ] ビジネス要件と技術要件の整合性
- [ ] スコープが明確か（何をやらないかも含めて）
- [ ] 主要リスクが識別されているか（セクション6参照）
- [ ] 予算・スケジュールの実現可能性

**成果物:** 要件定義書（Issue説明またはドキュメント）

---

### F2: 設計フェーズ

**品質の観点:** 設計整合性・デザイン資産の完備・技術選定の妥当性

**確認項目:**
- [ ] `docs/DESIGN-ASSET-INTAKE-CHECKLIST.md` の全項目がPASS
- [ ] `docs/DESIGN-SPEC.md` との整合性確認
- [ ] APIインターフェース・データ構造の設計レビュー完了
- [ ] 技術スタック（`docs/TECH-STACK.md`）との整合性
- [ ] セキュリティ要件の設計レベルでの考慮

**成果物:** 設計ドキュメント、インテークチェックリスト記入済み

---

### F3: 開発フェーズ

**品質の観点:** コード品質・単体テスト・レビュー

**確認項目:**
- [ ] コーディング規約の遵守（ESLint等の静的解析ゼロエラー）
- [ ] 単体テストの作成（`tests/` 配下、カバレッジ目標80%以上）
- [ ] プルリクエストごとのコードレビュー実施・承認
- [ ] `console.error` / 未処理例外ゼロ（本番コード）
- [ ] 依存パッケージの脆弱性チェック（`npm audit`）
- [ ] コメント・ドキュメントの更新

**成果物:** テスト済みコード、PRレビュー記録（GitHubのPR）

---

### F4: テストフェーズ

**品質の観点:** 統合・システム・回帰テストの網羅性・欠陥管理

**確認項目:**
- [ ] テスト計画書（`docs/TEST-PLAN-{機能名}.md`）の作成・承認
- [ ] 統合テストケースの実施（全ケース記録）
- [ ] スモークテスト通過（`scripts/smoke-test.sh`）
- [ ] パフォーマンステスト基準達成（`docs/PERFORMANCE-TESTING.md`参照）
- [ ] 発見された欠陥の全件Issueへの記録
- [ ] 重大・高優先度欠陥の修正と再テスト完了
- [ ] 回帰テストの実施（既存機能への影響確認）

**成果物:** テスト結果報告書、欠陥一覧（GitHubのIssues）

---

### F5: デプロイフェーズ

**品質の観点:** リリース前ゲート・デプロイ手順・ロールバック準備

**確認項目（セクション7の品質ゲート全項目）:**
- [ ] 本番デプロイ手順書（`docs/PRODUCTION-DEPLOYMENT.md`）の確認
- [ ] デプロイスクリプト（`scripts/deploy-prod.sh`）の動作確認
- [ ] 監視スクリプト（`scripts/monitor-production.sh`）の設定確認
- [ ] ロールバック手順の確認・テスト
- [ ] 本番環境でのスモークテスト実施

**成果物:** デプロイ記録、監視アラート設定

---

### F6: 保守フェーズ

**品質の観点:** 継続的モニタリング・インシデント対応・改善サイクル

**確認項目:**
- [ ] エラー率・パフォーマンスメトリクスの週次確認
- [ ] インシデント発生時の根本原因分析（RCA）の実施・文書化
- [ ] ユーザーフィードバックの収集・分類・対応計画
- [ ] 四半期ごとのSOP見直し（セクション2参照）

**成果物:** 週次メトリクスレポート、インシデント報告書（Issue）

## 5. レビュー→承認→品質確認→GitHub操作フロー

### 5.1 標準フロー図

```
[作業完了宣言]
     ↓
[セルフレビュー] ← 担当エージェント/メンバー
  - セクション4の該当フェーズチェックリスト確認
  - セクション7の品質ゲート確認（デプロイ時）
     ↓
[PRの作成（GitHub）]
  - ブランチ命名: feature/{issue-id}-{短い説明}
  - PR説明: 変更内容・テスト結果・関連Issue番号
  - reviewerのアサイン
     ↓
[コードレビュー] ← 開発リーダー（または上級エンジニア）
  - 指摘事項をPRコメントへ記録
  - 承認 OR 変更要求
     ↓ (変更要求の場合は修正後に再レビュー)
[テスト確認] ← QAリーダー
  - CIパイプラインのテスト結果確認
  - スモークテストの手動実行（必要時）
     ↓
[マージ承認]
  - GitHub上でPR承認（Approve）
  - Squash mergeまたはMerge commit（プロジェクト規約に従う）
     ↓
[本番デプロイ判定]（リリースの場合のみ）
  - セクション7の品質ゲート全項目確認
  - PM + QAリーダーによるリリース承認
     ↓
[デプロイ実行]
  - `scripts/deploy-prod.sh` 実行
  - 本番スモークテスト実施
  - 監視ダッシュボード確認（`scripts/monitor-production.sh`）
     ↓
[完了・Issueクローズ]
  - GitHubのIssueをクローズ
  - PaperclipのIssueをdoneに更新
```

### 5.2 GitHub操作ルール

| 操作 | ルール |
|------|--------|
| ブランチ作成 | `feature/{issue-id}-{説明}` / `fix/{issue-id}-{説明}` / `hotfix/{説明}` |
| コミットメッセージ | `{type}: {要約}` (type: feat/fix/docs/test/chore) |
| PR作成 | テンプレートに従う・関連Issueを `Closes #xxx` で紐付ける |
| マージ | レビュー承認後のみ。mainへの直接pushは禁止 |
| タグ | リリース時に `v{major}.{minor}.{patch}` のタグを付与 |

## 6. 品質保証活動と予防策（リスク対応）

予防策は「何が起きうるか」を具体的に想定し、事前対応策を定義することで初めて機能する。以下にUniteCubeで頻出するリスクカテゴリと具体的な対応策を示す。

### 6.1 リスクカテゴリと予防策一覧

| # | リスクカテゴリ | 具体的なリスク例 | 予防策 | 担当者 |
|---|--------------|----------------|--------|--------|
| R1 | 要件の不明確さ | 要件が曖昧でエージェントごとに解釈が異なる | 要件定義時にAcceptance Criteriaをリスト化してIssueに記載。着手前に全担当者がレビュー | PM |
| R2 | デザイン資産不足 | 実装途中でアイコン・スタイル定義が不足する | `docs/DESIGN-ASSET-INTAKE-CHECKLIST.md`によるインテーク完了（PASS）を開発着手の前提条件とする | デザインリーダー |
| R3 | コード品質劣化 | テストなし・レビューなしのコードが本番に入る | すべてのPRにコードレビュー必須・CIでの自動テスト実行を強制 | 開発リーダー |
| R4 | 回帰バグ | 新機能追加で既存機能が壊れる | 本番デプロイ前の回帰テスト実施・`tests/` 配下の既存テスト全件通過を必須とする | QAリーダー |
| R5 | 本番障害 | デプロイ後にエラーレートが急増する | `scripts/monitor-production.sh`による監視・アラート設定。ロールバック手順の事前確認 | 開発リーダー + PM |
| R6 | セキュリティ脆弱性 | 依存パッケージに既知の脆弱性が含まれる | `npm audit` をCIに組み込み・重大度 high 以上は即座に対応 | 開発リーダー |
| R7 | パフォーマンス低下 | ページロードタイムが基準を超える | デプロイ前にパフォーマンステスト実施（`docs/PERFORMANCE-TESTING.md`参照） | QAリーダー |
| R8 | エージェント間の作業衝突 | 複数エージェントが同一ファイルを並行変更する | Issueのcheckout制御を使用・ブランチ戦略の遵守・PRでのコンフリクト解消 | 全担当エージェント |

### 6.2 予防策の有効性確認

各リリース後に以下を確認し、予防策が実際に機能しているかを評価する。

- 各リスクカテゴリで「予防策が実施されたか」をリリースチェックリストに記録
- 不適合発生時（セクション8）に「どの予防策が機能しなかったか」を根本原因分析で特定
- 四半期レビューで予防策の有効性を評価し、必要に応じてSOP改訂（セクション2）

## 7. 品質確認ゲート チェックリスト

本番リリース前に、以下の全項目をPASSしなければならない。

### 7.1 コード品質ゲート

- [ ] すべての単体テストが通過（`npm test` でゼロ失敗）
- [ ] テストカバレッジが80%以上
- [ ] 静的解析（ESLint等）でゼロエラー
- [ ] コードレビュー承認済み（開発リーダーApprove）
- [ ] `npm audit` で高深刻度（high以上）の脆弱性ゼロ

### 7.2 機能品質ゲート

- [ ] すべての受入れ基準（Acceptance Criteria）を満たす
- [ ] スモークテスト通過（`scripts/smoke-test.sh`）
- [ ] 重大・高優先度の欠陥がゼロ（Issueがすべてcloseまたはwontfix）
- [ ] 回帰テスト通過（既存機能への影響なし）

### 7.3 パフォーマンス・セキュリティゲート

- [ ] パフォーマンスベンチマーク達成（`docs/PERFORMANCE-TESTING.md`の基準参照）
- [ ] セキュリティスキャン実施・重大問題なし

### 7.4 デプロイ準備ゲート

- [ ] ロールバック手順の確認完了
- [ ] 監視・アラート設定の確認（`scripts/monitor-production.sh`）
- [ ] PM + QAリーダーによるリリース承認

**判定基準:** 全項目PASSでリリース可。1項目でもFAILの場合はリリース禁止。

## 8. 不適合時の是正処置手順

### 8.1 不適合の定義と深刻度分類

| 深刻度 | 定義 | 例 | 対応期限 |
|--------|------|-----|---------|
| Critical | 本番サービス停止・データ損失・セキュリティ侵害 | 全ページ404・認証バイパス | 即時（2時間以内） |
| High | 主要機能の動作不能 | ナビゲーション崩壊・フォーム送信不可 | 24時間以内 |
| Medium | 部分的な機能障害・UXの大幅な劣化 | 特定ブラウザでのレイアウト崩れ | 次スプリント内 |
| Low | 軽微な表示の問題・改善要望 | 誤字・アイコンサイズのズレ | バックログ管理 |

### 8.2 是正処置フロー

```
[不適合の発見]
  - テスト中・モニタリング中・ユーザー報告
     ↓
[Issueの起票（GitHub / Paperclip）]
  - タイトル: [BUG] {深刻度} {現象の要約}
  - 内容: セクション9.2の欠陥報告書テンプレートに従う
  - 優先度: 深刻度に対応するpriority設定
     ↓
[深刻度の評価・優先度アサイン]
  - QAリーダーが評価・担当者アサイン
     ↓
[根本原因分析 (RCA)]
  - 「なぜ」を5回繰り返す（5 Whys法）
  - 再発防止策を特定
     ↓
[修正実施]
  - 担当エージェント/メンバーが修正
  - Criticalの場合: hotfixブランチから直接本番パッチ
     ↓
[再テスト・回帰テスト]
  - 修正箇所の確認テスト
  - 関連機能の回帰テスト
     ↓
[是正処置の確認]
  - QAリーダーが再テスト結果を確認・承認
     ↓
[再発防止策の反映]
  - テストケースの追加
  - 予防策一覧（セクション6）の更新
  - 必要に応じてSOPの改訂（セクション2）
     ↓
[Issueクローズ・記録]
  - 是正処置の内容をIssueコメントに記録
  - Issueをcloseし、根本原因・対応策をドキュメント化
```

### 8.3 エスカレーション基準

- **Critical** → 発見後即座にPM + 開発リーダーへ通知・`docs/BOARD-ESCALATION-GUIDE.md`に従い必要に応じてボードへ報告
- **High** → 24時間以内にQAリーダーとPMへ報告
- **本番障害** → `docs/PRODUCTION-DEPLOYMENT.md`のインシデント対応手順に従う

## 9. 標準文書テンプレート

エージェントおよびチームメンバーが品質管理文書を作成する際は、以下のテンプレートに従うこと。これにより、誰が作成しても同一の書式・内容レベルが保たれる。

### 9.1 テスト計画書テンプレート

ファイル名: `docs/TEST-PLAN-{機能名またはリリース番号}.md`

```markdown
# テスト計画書: {機能名またはリリース番号}

**作成日:** YYYY-MM-DD
**作成者:** {担当者名/エージェント名}
**対象Issue:** {Issue番号}
```

## 10. 品質メトリクスと改善サイクル

### 10.1 定常メトリクス（週次収集）

| メトリクス | 測定方法 | 目標値 | 責任者 |
|------------|----------|--------|--------|
| テストカバレッジ | `npm test -- --coverage` | 80%以上 | 開発リーダー |
| CIビルド成功率 | GitHub Actionsの成功/失敗数 | 95%以上 | 開発リーダー |
| 重大欠陥解決率 | GitHubのIssueラベル集計 | 100%（Critical/High） | QAリーダー |
| 本番エラーレート | `scripts/monitor-production.sh` | <0.1% | PM |
| デプロイ頻度 | 月当たりのリリース数（参考値） | - | PM |

### 10.2 リリース時メトリクス

- 品質ゲートの合格率（何項目でFAILしたか）
- リリースサイクルタイム（開発着手からリリースまでの日数）
- 本番障害発生率（リリース後1週間以内の重大欠陥数）

### 10.3 改善サイクル（PDCAサイクル）

```
Plan（計画）:
  四半期ごとに品質目標を設定（メトリクス目標値の見直し）
     ↓
Do（実施）:
  セクション4〜7の手順に従って品質活動を実施
     ↓
Check（確認）:
  週次メトリクスレポート（セクション9.3テンプレート）で実績を確認
  月次でトレンドを分析
     ↓
Act（改善）:
  目標未達のメトリクスについて根本原因分析
  予防策の追加・更新（セクション6）
  SOPの改訂（セクション2の改訂手順に従う）
```
