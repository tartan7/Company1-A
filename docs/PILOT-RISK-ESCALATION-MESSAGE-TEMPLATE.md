# Pilot Risk-Escalation Message Template (v1)

*Prepared Date: YYYY-MM-DD*
*Owner: CMO / Pilot Owner*
*Pilot: [Client Name / Pilot ID]*

## 1. Purpose
Use this template when a pilot risk crosses a predefined escalation threshold and stakeholders need a fast, structured, decision-ready message.

This format is designed for chat, email, and executive channels with minimal rewriting.

---

## 2. When To Escalate

Escalate immediately when one or more conditions are true:

- A `High` risk becomes `Active` and threatens pilot timeline, KPI, or client commitment.
- A blocker remains unresolved longer than the SLA in the communication cadence.
- A dependency owner misses a committed due date without an approved recovery plan.
- Additional leadership decision, budget, or cross-team intervention is required.

If uncertain, escalate early with a clear confidence statement instead of waiting for perfect data.

| Field | Example |
| :--- | :--- |

| Risk Title | CRM schema change breaks attribution mapping |
| Severity | High / Critical |
| Current Status | Open / At Risk / Blocked |

| Current Mitigation | What is already underway |
| Decision or Support Needed | Specific ask from escalation audience |
| Owner | Name + role |
| Due Date / Decision Deadline | YYYY-MM-DD HH:MM TZ |

---

## 4. Core Escalation Message (Copy/Paste)

`Escalation: [Risk ID] - [Risk Title]`

`Severity/Status: [High/Critical] - [Open/At Risk/Blocked]`

`Impact: [What is at risk and by when; include KPI, launch date, SLA, or commitment]`

`What changed: [Trigger event or new evidence in 1 sentence]`

`Current mitigation: [Actions already in progress + owner]`

`Support needed: [Decision/resource/intervention required]`

`Decision owner: [Name/Team]`

`Needed by: [YYYY-MM-DD HH:MM TZ]`

`Fallback plan if delayed: [Consequence + contingency]`

`Next update: [YYYY-MM-DD HH:MM TZ]`

---

## 5. Audience Variants

### A) Executive Escalation (30-second read)

`Escalation: [Risk ID] threatens [timeline/KPI/client commitment]. We need [specific decision/support] from [owner] by [date/time] to avoid [business impact]. Current mitigation is [action], and contingency is [fallback].`

### B) Client-Facing Escalation

`We are escalating risk [Risk ID]: [short title]. Current impact is [impact]. Mitigation actions are underway: [actions]. To protect [timeline/KPI], we need [decision/input] by [date/time]. We will provide the next update by [time].`

### C) Internal Delivery Channel

`Risk escalation opened for [Risk ID] ([severity]). Blocker: [blocker]. Owner: [name]. Support needed: [ask] by [date/time]. If delayed, expected impact: [impact].`

---

## 6. Escalation Quality Checklist

Confirm before sending:

1. Risk ID and owner match the risk register.
2. Impact is measurable (date, KPI, SLA, or scope line item).
3. Ask is explicit and assigned to a named decision owner.
4. Deadline and next update timestamp are included.
5. Fallback plan is stated if support is not provided in time.
